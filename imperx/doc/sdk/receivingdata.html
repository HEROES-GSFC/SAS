Receiving Data

Receiving Data from a GigE Vision Transmitter

A GigE Vision device (transmitter) transmits blocks of data (such as images) over a network using the GigE Vision Streaming Protocol (GVSP).

The eBUS SDK provides a means of receiving those packets as they come over the network and assembling them into an area of memory that is accessible to the calling application.

The software component that receives and assembles streaming packets into a usable block of memory is commonly referred to as a data receiver.

A filter driver called eBUS Universal Pro is bundled with the eBUS SDK and can optionally be applied to a network adapter in order to maximize streaming performance (bypassing the standard network stack to achieve lower CPU usage and better data throughput).

Depending on whether a filter driver is applied or not, the data receiver may exist in kernel space or user space.

PvStreamBase (and consequently its child classes PvStreamRaw and PvStream) provides an interface for configuring and receiving data from the data receiver.

Note that there are two different classes that can be instantiated to receive blocks (such as images or raw data) from a GigE Vision transmitter.

PvStreamRaw provides a raw C++ interface to all the data receiver properties.

PvStream provides access to a PvGenParameterArray containing all the data receiver properties.

Both PvStreamRaw and PvStream inherit from PvStreamBase which contains common functionality for opening a connection to the data receiver and obtaining data from it.

The process of receiving data from a GigE Vision transmitter involves queuing buffers into the data receiver ensuring that there are always buffers available for the incoming data. A PvStreamBase object is normally used in conjunction with a PvPipeline object which helps manage the process of allocating and pushing buffers into the data receiver as needed.

Opening a PvPipeline object allocates a pool of available buffers (configurable by calling PvPipeline::SetBufferCount and PvPipeline::SetBufferSize) and starts a thread that continually feeds buffers into the data receiver. Each time a new block (such as an image or raw data) becomes available, its associated buffer is placed in a queue of available buffers that can be retrieved by calling PvPipeline::RetrieveNextBuffer.

If the calling application cannot retrieve buffers at the rate that data is arriving, buffers are silently dropped. To obtain a count of dropped buffers, refer to the PipelineBlocksDropped property accessible from the stream.

To control a GigE Vision device (i.e. tell it when and where to transmit data), the eBUS SDK provides a class called PvDevice. For a complete working sample that connects to a GigE Vision device, opens a stream and tells the device when and where to begin streaming, refer to the sample entitled PvPipelineSample.

In some cases, it may be desirable for an application to manage the process of allocating and pushing buffers into the data receiver without making use of a PvPipeline object. For complete working samples that incorporate this concept, refer to PvStreamSample and PvCustomPipelineSample.
